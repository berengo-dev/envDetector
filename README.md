# env-doctor

**Validate your development environment before you start coding.**

`env-doctor` is a command-line tool that checks whether a developer's local environment matches a project's requirements. It verifies tools, environment variables, required files, and ports — catching issues before you waste time debugging.

```bash
cd my-project
env-doctor init --auto
env-doctor check
```

---

## The Problem

You clone a project, install dependencies, and then discover:
- Your runtime version is too old
- You're missing a required environment variable
- A critical tool isn't installed
- The default port is already in use

**env-doctor catches these issues in seconds.**

## Features

- **Auto-detection**: Scans project files and generates a configuration automatically
- **Tool version checking**: Validates installed binaries with flexible version matching
- **Environment variable verification**: Checks system variables and local configuration files
- **File existence checks**: Ensures required files are present
- **Port availability**: Verifies if ports are free or occupied
- **Colored terminal output**: Clear, readable status with visual indicators
- **Exit codes**: Returns 0 for success, 1 for failures — ideal for automated pipelines

## Auto-Detection

`env-doctor init --auto` detects your project type and generates the configuration automatically:

### Supported Manifests

`env-doctor` scans for standard project manifests and extracts requirements without hardcoding any specific technology stack:

- **JSON manifests**: Reads `package.json` and any JSON file containing dependency fields (like `dependencies`, `devDependencies`, or `engines`). Extracts tool versions and detects locally installed binaries.
- **Go modules**: Reads `go.mod` for the runtime version and checks for common tooling files (`Makefile`, `Dockerfile`).
- **Python manifests**: Reads `requirements.txt` or `pyproject.toml` to extract dependencies and runtime constraints.
- **Generic files**: Falls back to checking for `.env`, `.env.example`, `docker-compose.yml`, and other common project files.

The tool recognizes standard project manifests and extracts:
- Runtime versions and constraints
- Required development tools (only those with actual executable binaries)
- Environment variables from template files
- Common project files
- **Note**: Port checks are not auto-generated. You can add them manually if needed.

## Configuration Reference

### Configuration File Structure

```yaml
version: "1"

# Required binaries with expected versions
tools:
  runtime: "1.21"        # Prefix match: 1.21.x accepted
  service: "20.x"        # Wildcard match: any 20.x accepted
  container-tool: "24.x"  # Wildcard match: any 24.x accepted

# Required environment variables
env:
  - DATABASE_URL
  - API_KEY
  - SECRET_KEY

# Required files
files:
  - .env
  - config.json
  - README.md

# Port checks
ports:
  3000: free        # Must be available
  5432: occupied    # Must be listening
```

### Version Matching

Version checks run the tool with `--version` (falling back to `version`). The first semantic-looking version is extracted and compared:

- **Exact**: `1.21.5` matches only `1.21.5`
- **Prefix**: `1.21` matches `1.21.0`, `1.21.5`, etc.
- **Wildcard**: `24.x` matches any `24.*` version

### Environment Variables

`env-doctor` checks for environment variables in two places:

1. System environment variables
2. Project configuration files (if present)

If a variable is found in the local configuration but not in the system environment, it shows: `set (from .env)`.

### Local Tool Detection

For projects with local dependencies, `env-doctor` checks for binaries in local bin directories (such as `node_modules/.bin/` or `.venv/bin/`) before checking the system PATH. This means it detects locally installed tools without requiring global installation.

The tool only reports dependencies that actually expose executable binaries, avoiding pure libraries that cannot be run as commands.

## Use Cases

### 1. New Team Member Onboarding

```bash
cd project
env-doctor check
# "You need Node 20, you have 18. Update your version manager."
```

### 2. Machine Switching

```bash
cd project
env-doctor check
# "Missing environment variable: DATABASE_URL. Check .env.example"
```

### 3. CI/CD Pipeline

```yaml
# .github/workflows/ci.yml
- name: Validate Environment
  run: env-doctor check
  # Fails the pipeline if environment is misconfigured
```

### 4. After Pulling Changes

```bash
git pull origin main
env-doctor check
# "New dependency added: eslint. Install with your package manager."
```
